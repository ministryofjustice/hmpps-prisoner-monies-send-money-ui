# ADR-0001: Handling of Start and Informational Pages in Non-Production Environments

## Status
Accepted

## Date
2026-06-15

## Context

As part of the ongoing migration effort, we are performing a strict lift-and-shift of the existing Python/Django application into a new architecture consisting of a TypeScript/Express frontend and Kotlin backend services. The goal of this phase is to replicate existing behaviour without introducing functional changes, deferring improvements to a later phase.

During analysis of the existing system and its deployment model, it was identified that certain UI pages—specifically the **start page** and associated informational pages—are handled differently across environments:

- In **production**, these pages are not served by the application itself. Instead, requests are redirected to externally managed endpoints.
- In **development, test, and pre-production environments**, these pages are served directly by the application.

This behaviour is currently replicated in the new service using a configuration-driven request handler:

```typescript
export type startPageHandlerConfig = {
  production: boolean
  productionStartPageUrl: string
  sendMoneyUrl: string
}

function startPageHandler(config: startPageHandlerConfig): RequestHandler {
  return (_req, res) => {
    if (config.production) {
      return res.redirect(config.productionStartPageUrl)
    }

    return res.render('pages/start-page', { sendMoneyUrl: config.sendMoneyUrl })
  }
}
```

The implication of this setup is that content rendered in non-production environments is **not representative of what end users ultimately see in production**.

The team must decide the level of effort to invest in replicating, validating, and testing the content of these pages in non-production environments during the lift-and-shift phase.

## Decision

We will **not attempt to fully replicate or validate the content of start and informational pages in non-production environments**.

Instead:

- These pages will be implemented with minimal content, ensuring:
  - The page exists
  - Core structural elements are present (e.g. header)
  - Required user interactions function (e.g. "Start Now" button)
- Testing in non-production environments will focus on:
  - Routing behaviour
  - Correct rendering invocation
  - Functional navigation (e.g. correct links and actions)
- Content accuracy, styling fidelity, and full UX parity **will not be tested or validated** in non-production environments.

In production, the application will **continue to redirect to externally managed pages**, and those pages remain the source of truth for user-facing content.

## Consequences

### Positive

- **Reduced delivery time**: Eliminates time spent recreating, testing and validating content that will not be used in production.
- **Improved focus**: Allows the team to prioritise functional equivalence and backend/frontend integration rather than caring about static content on specific pages.
- **Alignment with lift-and-shift objective**: Avoids unintended divergence or enhancement during migration.

### Negative

- **Reduced UI parity**: Non-production environments will not visually match production for these pages.
- **Confusion for developers**: Differences between environments must be clearly understood and documented. It should be noted that this has already caused confusion in the team
- **Limited control over these pages in Production**: Accessibility checks on these pages in non-production environments will be minimal and may not reflect production behaviour. We would be reliant on the owning team to perform these checks. Additionally, we would be reliant on an external team to ensure that their pages link correctly to our service once we go fully live and switch over.
- **Deviation from the 'pure lift and shift' plan** - The existing 'old world' services include some content in their dev / test pages, although this static content no longer reflects what is live in Production

### Neutral / Mitigations

- This limitation is **explicitly documented in this ADR** and should be referenced in test strategies.
- End-to-end tests (e.g. Playwright) should:
  - Validate redirection behaviour in production-like configurations
  - Avoid asserting on page content for these routes
- Any issues related to production page content must be addressed in the system that owns those pages.

## Alternatives Considered

### 1. Fully replicate production content in non-production environments

- **Rejected**: This would require duplicating externally managed content and validating it unnecessarily.
- **Reason**: Does not align with lift-and-shift goals and introduces wasted effort.

### 2. Omit these pages entirely in non-production environments

- **Rejected**: Would break application flow and prevent functional testing.
- **Reason**: Minimal implementations are required to preserve navigation and flow.

## Implementation Notes

- Ensure all relevant handlers follow the same pattern as `startPageHandler`.
- Clearly document environment-specific behaviour in README or developer documentation.
- Avoid asserting content in tests; prefer behavioural and routing assertions.
- Maintain consistency across environments using configuration flags.

## Future considerations

These may not be the only pages covered by an external team in production.

It would be prudent to do some research as a team to find if there are any other pages that fall under the purview of other teams before development begins on related features.
